# パネル機構・インシデント機構のブラッシュアップ

## Context

ユーザーから「グローバル/兄弟プロジェクトの各種パネル（多視点レビュー機構）にブラッシュアップの余地がないか」「インシデント記録にも無駄が多いので改善したい」との依頼を受けた。アーキテクチャ判断・トレードオフ評価に該当するため、実装前にFableサブエージェントへ2回設計批評を依頼済み（`claude-md-panel`／`spec-panel`のパネル批評、`log-incident`のインシデント機構批評）。両批評とも「実装はまだ、指摘のみ」で完了しており、本計画はその指摘のうち**優先度「高」の項目のみ**を対象にする（中・低優先度は今回見送り、理由は各節に記載）。

対象は以下の2系統。いずれも `~/.claude/` 配下のグローバル設定（全プロジェクト共通）と、claude-container プロジェクト自身の `.claude/` 配下。findsummits の `spec-panel`／`spec-panel-gate.sh` 自体への修正は対象外（別リポジトリであり、パネル批評では「参考にすべき優れた設計」として引用したのみで、findsummits側の変更は別セッションの管轄）。

## 対象1: claude-md-panel — 強制方式を成果物鮮度ゲートへ統一

**現状の問題**（Fable批評より）: `claude-md-panel-mark.sh`（PostToolUse）が `/tmp/claude-md-panel-pending-${sid}-${key}` を作成し、`claude-md-panel-stop.sh`（Stop hook）がその**存在**だけを見て未レビューを判定する。解除はコマンド手順6が `key=$(printf '%s' "ABS" | md5sum | cut -c1-16)` を独自に再現して `rm` する方式で、パス文字列が1文字でもズレる（相対/絶対、シンボリックリンク等）と解除に失敗し Stop hook が最大3回空回りする。かつキー生成ロジックが hook 側とコマンド側の2箇所に重複しており、変更時に drift するリスクがある。

**方針**: keyのハッシュ生成ロジックを共有スクリプトに一本化し、解除条件を「rmコマンドの再現」から「レビュー成果物ファイルをWriteしたか（鮮度判定）」に変える。spec-panelの「成果物ベースのゲート」に倣う。

### 変更ファイル

1. **新規** `~/.claude/hooks/claude-md-panel-keyhash.sh`
   - 引数の絶対パスから `md5sum | cut -c1-16` でキーを算出して標準出力するだけの1行スクリプト。mark.sh・stop.sh・コマンド手順6の3箇所が共通利用する唯一の真実源にする。

2. `~/.claude/hooks/claude-md-panel-mark.sh`
   - `key=$(printf '%s' "$f" | md5sum | cut -c1-16)` を `key=$(bash "$(dirname "$0")/claude-md-panel-keyhash.sh" "$f")` に置き換え。
   - 催促メッセージを「今すぐ実行してください」から「このターン内の関連編集が全て完了してから実行してください」に修正（lessons: 連続編集は完了後にまとめてレビューする方針との整合。編集途中の中間状態への指摘を避ける）。

3. `~/.claude/hooks/claude-md-panel-stop.sh`
   - 判定を「pendingファイルの存在」のみから、「pendingファイルが指す対象パスについて、`~/.claude/panel-findings/${key}.md` が存在し、かつその mtime が対象CLAUDE.mdファイルの mtime 以降か」に変更。findings が対象編集より新しければレビュー済みとみなし pending/review マーカーを消して通す。無ければ従来どおり最大3回 block。
   - block時のメッセージを「rmしてください」から「`~/.claude/panel-findings/${key}.md` にレビュー内容を Write してください」に変更。

4. `~/.claude/commands/claude-md-panel.md`
   - 手順6を書き換え: rmコマンドの提示をやめ、「`key=$(bash ~/.claude/hooks/claude-md-panel-keyhash.sh "ABS")` を算出し、`~/.claude/panel-findings/${key}.md` へ手順3の指摘表と手順4の総評をそのまま Write する」に変更。Write自体が解除条件になるため rm操作は不要になる。

**見送る指摘**（中・低優先度、今回対象外): 出力フォーマットへの「仕様検討/作業実行」2セクション分割・裏取り済み列の追加（spec-panelとの相互移植）。

## 対象2: インシデント機構（log-incident）— A/B型分岐・既知パターン台帳・状態行ルールの是正

**現状の問題**（Fable批評より、claude-container実データで検証済み）:
- 全5件のインシデントのうち4件（1,2,4,5番）が「ホストクラッシュ→原因はカーネル層の疑い→プロジェクトスコープ外→対策反映なし」というほぼ同一の結論を、フルフォーマット（セッションID〜対策反映まで7項目）で毎回繰り返している。
- ホストフリーズ系は前セッション断絶後の**汚染していない新セッション**による事後記録なのに、現行の「段階4: セッション隔離」が一律に「作業継続禁止・セッション終了要求」を求めている。保全すべき汚染セッションが存在しないケースにまで同じ儀式を強制している。
- スキル自身が明記する「状態行は1本のみ・書換えのみ（追記禁止）」というルールを、実データで**2/5件が実際に違反**している（`2026-07-04_1139`はbullet形式の状態行が2本併存、`2026-07-02_2252`は見出し形式`### 状態:`とbullet形式`- 状態:`が併存 — 後者はFableサブエージェントが実ファイル裏取りで新規発見）。session-start.shの`tail -1`判定がたまたま両方とも正しい方向に転んでいるため自動判定は壊れていないが、運用が構造的に逸脱しやすいルールになっている。

**方針**: 判定基準は「原因のスコープ」ではなく「**現行セッションの汚染疑いの有無**」にする（記録時点で機械的に判定できるため。原因ベースの分岐は記録時点でまだ確定していないことが多い — 実際`2026-07-02_2252`も当初は原因不明だった）。

### 変更ファイル

1. `~/.claude/commands/log-incident.md`
   - **トリガー節の直後に分岐基準を追加**: 異常が**現行セッション内で発生・観測された**（自走・ツール不調・FS変化等、トリガー節の例に一致）→ **A型**（現行どおりフル記録＋段階4のセッション隔離を維持）。異常は**前セッションの断絶によるもので、現行セッションはそれを検知・調査するために起動した新セッション**（例: ホストフリーズ・強制再起動後に起動した新セッション）→ **B型**（軽量記録。段階4のセッション隔離は不要、作業継続可）。
   - **B型の軽量フォーマット**: 必須項目を「発生時刻（不明可）・症状1〜2行・実害確認結果（git status/podman ps等の裏取り結果）・状態・対策反映」に絞る（セッションID・観測出力の詳細等は任意）。
   - **既知パターン台帳の運用ルールを追加**: 同一症状（例: ホストクラッシュ／カーネル層疑い／プロジェクトスコープ外）が既に一度フル記録・原因究明を経て「これ以上の追跡は費用対効果が低い」と合意済みの場合、プロジェクトの `.claude/incidents/known-patterns.md`（無ければ新規作成）にパターンごとの節を作り、アンカーインシデント（最初にそのパターンを確立したフル記録ファイル）へのリンク＋再発ログ（1行/回: 日時・症状一言・実害確認結果）を追記する運用に切り替えてよい、と明記。**エスカレーション条件**: (a) 同一パターンの再発ログが3回を超えた場合、(b) 実害が新たに確認された場合、(c) 従来と異なる新しい原因情報が判明した場合（例: 具体的なカーネルバグの特定）は、軽量追記をやめてフルインシデントとして新規ファイル化する。
   - **状態行ルールを実運用に合わせて修正**: 「書換えのみ・追記禁止」を「**末尾追記のみ・複数存在する場合は最後の状態行が正**」に変更する（session-start.shの`tail -1`判定は元々この方式を前提にしており、ルール文言の方が実装と乖離していた）。ただし「同一ファイル内に状態を示す行を複数の異なる表記（見出し形式`### 状態:`とbullet形式`- 状態:`の併存等）で書かない」ことは明記して残す（表記ゆれによる曖昧さの防止）。

2. `.claude/hooks/session-start.sh`（claude-container自身）
   - 既存の未解決判定ループ（26-35行目）に、状態行の**複数併存を検知する広域正規表現**を追加する。狭域（既存、未解決判定用）: `^\s*[-*]?\s*\*{0,2}状態\*{0,2}\s*[:：]`。広域（新規、hygiene警告用、見出し形式`###`も拾う）: `^\s*[-*#]*\s*\*{0,2}状態\*{0,2}\s*[:：]`。広域正規表現でのマッチ数が2以上のファイルを`MULTI_STATUS_LIST`に集約し、既存の`UNRESOLVED_COUNT`ブロックと同様の形式でPass2に出力する（📋未解決インシデントのブロックの直後に「⚠️ 状態表明が複数箇所にあるインシデント: N件」を追加。表記の曖昧さを次セッションに引き継がないための警告であり、ブロックはしない）。
   - `for f in "$ROOT"/.claude/incidents/*.md` のglobを `for f in "$ROOT"/.claude/incidents/[0-9]*.md` に変更する（ファイル名は`YYYY-MM-DD_...`形式のみという`log-incident.md`の既存規約を反映。新設する`known-patterns.md`のような非インシデントファイルが将来このディレクトリに増えても誤って「状態行なし＝未解決」と誤検知しないようにする構造的な対策）。

3. **新規** `.claude/incidents/known-patterns.md`（claude-container）
   - 既存の「ホストクラッシュ/フリーズ（原因: ホストカーネル層の不安定性、プロジェクトスコープ外）」パターンを1件登録する。アンカーは `2026-07-02_2252_crun-kill-failed-on-session-exit.md`（唯一原因を`psi_avgs_work` Oopsまで特定できた記録）とし、`2026-07-02_1535`・`2026-07-02_1916`・`2026-07-03_1022`・`2026-07-04_1139`の4件を過去の再発ログとして遡って1行ずつ記載する（今後はこの4件のような新規フル記録ではなく、本台帳への追記で足りることを示す運用開始の起点とする）。

4. 既存インシデント2件の状態行重複を是正（今回発見した違反の是正、ルール変更と整合させる）:
   - `.claude/incidents/2026-07-02_2252_crun-kill-failed-on-session-exit.md`: 見出し形式`### 状態: 原因確定・環境側はホスト再起動待ち`を本文中の説明文に変換し（情報は失わない）、bullet形式`- 状態: 解決済（...）`のみを状態行として残す。
   - `.claude/incidents/2026-07-04_1139_host-freeze-during-research-request.md`: 2本の`- 状態:`行を1本に統合する（`解決済（恒久対策: なし、原因はfutex spinlockデッドロックとjournalctlで確定）`のように、2本目の追加情報を1本目に統合)。

**見送る指摘**（中優先度、今回対象外）: 7項目の必須/任意の仕分けをB型フォーマットの厳密な表化まで作り込むこと（今回は上記の簡易な絞り込みに留める）。

## 検証

- **claude-md-panel**: 実際のhookイベントを介さず、JSON入力を模擬してシェルスクリプト単体で確認する。

  2. findingsファイル未作成の状態で `echo '{"session_id":"test123","stop_hook_active":true}' | bash claude-md-panel-stop.sh` → block判定を確認

- **インシデント機構**:
  1. 新しい広域正規表現を既存5ファイルに対し`grep -cE`で手動実行し、`2026-07-02_2252`と`2026-07-04_1139`の2件のみが2件以上ヒットすることを確認（他3件は1件ヒット）
  2. 2件の状態行是正後、同じ正規表現を再実行し警告が解消されることを確認

- 全変更後、`git status`/`git diff`で意図した差分のみになっていることを確認してからコミット（グローバル`~/.claude/`側は別リポジトリのため、claude-container側の変更のみこのプロジェクトでコミットし、グローバル側の扱いはユーザーに確認する）
