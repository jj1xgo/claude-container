# Best Practices

> `lessons.md` の 17 項目を蒸留した高レベル原則集。
> 詳細な教訓・事例は `lessons.md` を参照。

## 1. 【合意と保全を先に】

**何をするか**: 仕様・設定の変更は必ずユーザーの合意を得てから行う。全プロジェクト共通で読まれる共有設定ファイル（`~/.claude/CLAUDE.md` 等）を編集する前には、元の内容を保全してから上書きする。

**なぜ有効か**: 合意と保全を徹底するようにしたら、無断の仕様変更や、共有ファイル上書きによる他プロジェクト設定の消失事故（2026-04-25）を防げるようになった。

**よくある失敗パターン**: 技術的困難を理由に黙って仕様を変える、あるいは全プロジェクト共通ファイルへプロジェクト固有の内容を書き込んで元の内容を消失させる。

## 2. 【許可範囲を超えない】

**何をするか**: `dangerouslyDisableSandbox: true` を要するようなコマンドや、許可されていないパスへの書き込みを実行しない。範囲外が必要になったら理由を説明しユーザーに許可を得る。

**なぜ有効か**: 許可範囲を先に確認するようにしたら、プロジェクトの意図しない領域への書き込みという重大な技術者倫理違反を未然に防げるようになった。

**よくある失敗パターン**: サンドボックス制約を「作業を進めるための障害」とみなし、確認なく無効化・迂回してしまう。

## 3. 【指示範囲で止まる】

**何をするか**: 「〜まで」「〜だけ」と範囲が明示された指示は、その境界で必ず作業を止める。次のステップが明らかでも、明示的な指示なしに先へ進まない。

**なぜ有効か**: 境界で止まるようにしたら、ユーザーが確認・タイミング調整のために介入する余地を奪わずに済むようになった。

**よくある失敗パターン**: 「どうせ次もやるから」と先読みして指示範囲を超えて作業を進めてしまう。

## 4. 【前提は実機で確認する】

**何をするか**: パッケージの有無・ビルド結果・ドキュメントの記述・handover の「完了」記述など、推測や過去の記録だけで真偽を断定せず、`dpkg -l` や実行結果など実機の裏取りで締める。

**なぜ有効か**: 実機確認を徹底するようにしたら、誤った前提記録（例:「ca-certificates は存在する」という誤記）を引き継いで手戻りする事故を防げるようになった。

**よくある失敗パターン**: handover やログからの推測だけで「存在する/しない」「完了した」を断定し、後で実態と食い違っていたことが判明する。

## 5. 【ビルドキャッシュは文字列一致で判定されると知る】

**何をするか**: Docker/Podman のビルドキャッシュは命令文字列の一致で判定される点を前提に設計する。キャッシュを意図的に破棄したい箇所は `ARG` を宣言するだけでなく `RUN` 内で実際に参照する。

**なぜ有効か**: この前提を踏まえて設計するようにしたら、「`-b` を付けても最新化されない」といったキャッシュ起因の見えないバグ（2026-06-11）を再発させずに済むようになった。

**よくある失敗パターン**: `ARG` を宣言しただけでキャッシュが破棄されると誤解し、中身が更新されないレイヤーに気づかない。

## 6. 【fail-closedを既定にする】

**何をするか**: ビルドと起動のような複数ステップを1コマンドに束ねない。失敗時に古い状態へフォールバックする構成（fail-open）を避け、失敗を検知したら後続処理へ進ませない構成（fail-closed）にする。

**なぜ有効か**: build と run を分離するようにしたら、「ビルド失敗に気づかず古いイメージで起動してしまう」事故（2026-07-02）を構造的に防げるようになった。

**よくある失敗パターン**: `podman compose run --build` のような一体化コマンドに頼り、失敗検知をユーザーの目視だけに委ねてしまう。

## 7. 【外部依存はビルド時スナップショット化する】

**何をするか**: レート制限のある外部 API への「起動のたびのライブ取得」を設計に組み込まない。変更頻度の低いデータはビルド時に1回だけ取得しスナップショットとして焼き込む。

**なぜ有効か**: ライブ取得をスナップショット化に変えるようにしたら、レート制限枯渇による fail-closed 起動不能という実障害（2026-07-02）を防げるようになった。レート制限カウンタ自体も分散エッジで揺れて信頼できないため、原因追及より依存を減らす設計変更の方が確実である。

**よくある失敗パターン**: 「毎回最新を取得すべき」という直感で設計し、ビルド時＋起動時の二重消費でクォータを枯渇させる。

## 8. 【コンテナ基盤の非互換性を疑う】

**何をするか**: Docker 前提の知識をそのまま Podman に当てはめない。ディレクトリマウント済みパスへのネストマウント等、挙動が異なる可能性を先に調査してから設計する。

**なぜ有効か**: 差分を先に疑うようにしたら、「サブパスだけ再マウントしても効果がない」といった非互換に起因する手戻りを防げるようになった。

**よくある失敗パターン**: Docker と同じ感覚でマウント構成を組み、Podman では無視される設定に気づかず時間を浪費する。

## 9. 【利用者向け変更はREADME更新を計画に載せる】

**何をするか**: 「ドキュメントを勝手に生成・変更しない」ルールを「更新しなくてよい」と拡大解釈しない。利用者に見える挙動が変わる変更では、README 更新をタスク候補として計画に含め、承認時にユーザーへ要否を確認する。

**なぜ有効か**: 更新候補として明示するようにしたら、無断変更を避けつつも更新漏れによる乖離を防げるようになった。

**よくある失敗パターン**: 「勝手に変更しない」を「対象外」と解釈し、計画段階で README 更新の判断機会自体を消してしまう。

## 10. 【ドキュメントの記述は定期的に実態と突き合わせる】

**何をするか**: 「git 管理対象/管理外」等のドキュメント上の記述は `git status`/`git ls-files` で裏取りする。README は実装ほど頻繁に更新されないため、区切りのタイミングで最終更新コミット以降の差分を横断確認する。

**なぜ有効か**: 定期的な突き合わせを習慣にするようにしたら、ドキュメントと実態の乖離（管理対象の思い込み違い、README の反映漏れ5箇所等）を早期に発見できるようになった。

**よくある失敗パターン**: ドキュメントの記述をそのまま信じ、実際の追跡状態やコード変更との乖離に気づかないまま作業を進める。

## 11. 【検証はフォールバック経路も含める】

**何をするか**: プロジェクト固有設定が「ある」経路だけでなく、「ない（既定の空設定にフォールバックする）」経路もビルド検証の対象に含める。

**なぜ有効か**: 両経路を検証対象にするようにしたら、通常の検証では通らないフォールバック経路特有の失敗（`pip3` 未同梱によるビルド失敗等）を事前に検出できるようになった。

**よくある失敗パターン**: 主要な利用シナリオ（プロジェクト側指定あり）だけを検証し、空設定側のビルド失敗に気づかない。

## 12. 【連続編集はまとめてレビューする】

**何をするか**: 同一ファイルへの連続編集が続く場合、都度レビューを走らせず、全編集が完了してから1回だけ整合性レビューを実行する。

**なぜ有効か**: 完了後にまとめてレビューするようにしたら、編集途中の中間状態に対する的外れな指摘を防げるようになった。

**よくある失敗パターン**: 編集の都度レビューを実行し、まだ書きかけの内容に対して誤った指摘を受けてしまう。
