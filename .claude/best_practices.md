## 1. 【共有領域は波及前提で扱う】

**何をするか**: 全プロジェクト共通で読まれる共有ファイル（`~/.claude/CLAUDE.md`・グローバルコマンド・bind mount された `~/.claude/settings.json`）への変更は、ホストと全プロジェクトのコンテナセッションへ一律波及する前提で扱う。プロジェクト固有の内容を書かず、編集前に元の内容を保全し、限定スコープで効かせたい設定は共有ファイルの直接編集でなく別ファイル＋明示指定（`claude --settings <path>` 等）で分離する。「次回検討」等のフラグは、どの仕組みにも拾われない場所に残さず、拾われる場所に置くかその場で処理する。

**なぜ有効か**: 波及範囲と保全・分離を意識するようにしたら、共有ファイル上書きによる他プロジェクト設定の消失事故（2026-04-25）、ホスト限定のつもりの設定が全コンテナへ波及する事故（2026-07-07 に構造を確認して回避）、放置フラグの複数セッションにわたる回収漏れ（2026-07-05）を防げるようになった。

**よくある失敗パターン**: グローバル領域にプロジェクト固有の手順を置く、bind mount された共有設定を「このプロジェクトだけの設定」のつもりで書き換える、「次回やる」メモを自動で拾われると思い込んで放置する。

## 3. 【前提は実機・実経路で確認する】

**何をするか**: パッケージの有無・「完了」「実害なし」などの判断は、推測・handover・ログの記述でなく実機の裏取り（`dpkg -l`・実行結果・`podman ps`・`git status`/`git diff`）で締める。外部サービスの API 仕様・画面仕様も記憶で断定せず、エラーメッセージが返す一覧や実際の画面で確認する。対になるコマンド間（`create`/`edit` 等）でフラグの対称性を類推しない。「Already logged in」型の成功風メッセージは意図した状態変化（トークン更新等）を保証しない。テキストコピペで失われる状態情報（UI のチェックボックスのオン/オフ等）から状態を断定せず明示的に尋ね、ユーザーの記憶する「以前の運用」と実ファイルが食い違ったら `git log` で経緯を確認してから対応を判断する。検証手段そのものの落とし穴（パイプで終了コードが変わる、`git ls-files` は未追跡を含まない、`bash -n` はヒアドキュメント破損を検知しない等）も疑う。検証は本体ロジックを実際に通す経路で行い（模倣コピーの自己チェックはトートロジー）、検知ロジックの変更は既存の実データ全件に走らせて確かめる。新しい検証ロジックは、手元にある導入前の壊れた実例で真っ先に実証する。転記・照合の完了確認は目視の突き合わせでなく `grep` 等の機械的な存在確認で締め、handover 等の自己申告記録も「書いた」で終えず直後に対象ファイルを grep で実地照合する。「複数経路で一致した」という検証も、透過的 DNS フィルタのように全経路が同一の中間層を通っていないか（独立性が本物か）を疑う。情報源の異なる出力の分離確認はグローバル best_practices #10 を参照。

**なぜ有効か**: 実機・実経路での確認を徹底するようにしたら、誤った前提記録の引き継ぎ、「実害なし」とクローズした障害の継続見落とし（2026-07-02）、lint 終了コードの誤認（2026-07-03）、画面仕様の誤案内や既存データの検知漏れ（2026-07-04）、転記済み確認の見落としや「複数リゾルバで確認済み」による外部要因の誤断定（2026-07-05〜06）、コピペ由来の状態誤認や記憶ベースの対応判断（2026-07-07）を防げるようになり、コマンド類推による動かない手順の混入や handover 自己申告と実態の乖離も検出できるようになった（2026-07-08〜09）。

**よくある失敗パターン**: handover やログ・記憶からの推測で「存在する/しない」を断定する、本体を通さない模倣テストで「検証済み」とする、見かけ上独立な複数経路の検証を独立と思い込む、自分が直前に書いた記録を照合なしで正とする。

## 4. 【ビルド資源の罠を知る】

**何をするか**: Docker/Podman のビルドキャッシュは命令文字列の一致で判定される前提で設計し、キャッシュ破棄したい箇所は `ARG` 宣言だけでなく `RUN` 内で実際に参照する。イメージ名・ステージング先のような共有ビルド資源はプロジェクトごとに名前空間で分離する。「ソース側設定の確認」と「実際に使われた成果物（ステージング先）の確認」は別工程として両方裏取りする。`Using cache` は入力が前回と同じという事実のみで、その入力が意図どおりであることは何も保証しない。

**なぜ有効か**: これらを前提に設計・検証するようにしたら、「`-b` を付けても最新化されない」バグ（2026-06-11）、複数プロジェクト交互ビルドの無言上書き（2026-07-02）、空フォールバックが安定してキャッシュヒットし続ける見落としを防げるようになった。

**よくある失敗パターン**: `ARG` 宣言だけでキャッシュが破棄されると誤解する、ソース側の設定ファイル確認だけで「ビルドに反映済み」と判断する。

## 5. 【fail-closedを既定にする】

**何をするか**: ビルドと起動のような複数ステップを1コマンドに束ねない。失敗時に古い状態へフォールバックする構成（fail-open）を避け、失敗を検知したら後続処理へ進ませない構成（fail-closed）にする。ただし「何で閉じるか」まで設計する: 一時的な失敗（タイムアウト・到達不能）と恒久的な状態変化（NXDOMAIN 等）を区別し、恒久条件でゲートが永久に閉じ続けない逃げ道を用意する。`set -euo pipefail` 下の `$(cmd1 | cmd2)` 代入は cmd1 の失敗で後続の判定に到達せず黙って中断するため、失敗しうるコマンドは `|| true` で明示的に握ってから結果を判定する。

**なぜ有効か**: build と run を分離するようにしたら、「ビルド失敗に気づかず古いイメージで起動してしまう」事故（2026-07-02）を構造的に防げるようになった。失敗の種別を区別するようにしたら、許可ドメインの恒久 NXDOMAIN 化で起動ゲートが永久に塞がる障害（2026-07-06）を解消できた。

**よくある失敗パターン**: `podman compose run --build` のような一体化コマンドに頼り失敗検知をユーザーの目視だけに委ねる、空文字判定だけでエラーハンドリングした気になりコマンド自体が失敗する経路を見落とす。

## 6. 【取得戦略は変化頻度で選ぶ】

**何をするか**: 外部データの取得は変化頻度と通知機構の有無に合わせて設計する。変更頻度が低くレート制限のあるデータは「起動のたびのライブ取得」にせず、ビルド時に1回だけ取得してスナップショット化する。TTL が短くセットごと切り替わるデータ（CDN の A レコード等）は起動時1回の解決に頼らず、ランタイムで差分追従する。プッシュ通知が構造的に飛ばない情報（自分の活動由来の更新等）はポーリング型の確認を設計する。

**なぜ有効か**: 頻度と機構に合わせて戦略を分けるようにしたら、レート制限枯渇による起動不能（2026-07-02）と、IP ローテーション後に新規接続が全滅する障害（同日）の両方を防げるようになった。レート制限カウンタ自体も分散エッジで揺れて信頼できないため、原因追及より依存を減らす設計変更の方が確実である。

**よくある失敗パターン**: 「毎回最新を取得すべき」「起動時に1回取れば十分」「更新は通知される」という一律の直感で設計し、クォータ枯渇や許可リスト全滅・見落としを招く。

## 7. 【境界はレイヤ差を疑う】

**何をするか**: コンテナ・ネットワーク境界の設計では、あるレイヤの対策・知識が別レイヤの挙動をカバーすると思い込まない。Podman は Docker と非互換がある（ネストマウント無視等）。アプリを PID1 に置くと reap 責務の欠如でゾンビが蓄積する。パケット DROP だけでは接続候補の選定（Happy Eyeballs の IPv6 試行等）は防げず、「候補に挙げさせない」対策との両輪が要る。役割の異なる防御レイヤ（LLM ベースの意味論的ゲートと OS/ネットワークレベルの機械的な壁、permission mode と sandbox 等）は独立した層であり、片方が賢く機能しても他方が不要になるわけではない。同一機能の中でも読み取りと書き込みで既定の制限範囲が非対称なことがある（sandbox は書き込みが狭く読み取りはほぼ全域）。ただし「独立」と整理した層にも結合点がありうる（ツール許可ルールのパスが sandbox のファイルシステム境界へマージされる等）ため、独立/結合の見取り図は公式ドキュメントで裏取りする。設定にも優先レイヤがあり（git config の local > global 等）、上位を差し替えても下位が勝つ経路を見落とさない。逆に、下位レイヤの UI が返す「変更できない」エラーは上位レイヤで反映済みという成功のサインでありうるため、標準の上書き元を順に切り分けてから解釈する。境界の見え方自体もレイヤ産物でありうる（sandbox 有効時の Bash からは書き込み禁止パスがデバイスファイルにマスクされて見える等）ため、ファイル種別の急変・大量未追跡は断定前に `mount` 等で機構を確認する。そもそも自分がどの境界の内側で動いているか（コンテナ内かホスト直か。`/home/node` の有無等で判別）を実機で確かめてから境界の実効性を判断し、同名・同種の設定変数でもホスト起動スクリプトだけが読むものとコンテナ内へ export されるものの配線差を確認する。同じ根本原因の問題にも自分の層で直せるものと製品仕様の層にあり直せないものが混在するため、対応可否は層ごとに切り分ける。

**なぜ有効か**: レイヤ差を先に疑うようにしたら、効かないマウント設定への時間浪費や、ゾンビ蓄積・間欠的な接続停止（2026-07-02）を多層の対策で塞げるようになり、「マウントしたのに反映されない」型の優先順位の見落とし（2026-07-05）も説明・文書化でき、sandbox 導入時も2層の独立・読み書きの非対称・優先度起因のエラー・マスク表示を正しく解釈でき（2026-07-07）、コンテナ内外での境界実効性の差や設定変数の配線差にも迷わなくなった（2026-07-08）。

**よくある失敗パターン**: 「DROP しているから遮断済み」「Docker で動いたから Podman でも同じ」と単層の対策・知識で完結したと判断する、片方の防御が的確だったからと他方を不要とみなす、UI のエラー表示を即「設定失敗」と解釈する、コンテナ前提の境界がホスト直実行でも効いていると思い込む。

## 8. 【汎用化は移行まで見届ける】

**何をするか**: 仕組みの移植・汎用化・有効化では、文言でなく実装詳細と移植先の制約・規模まで確認して合わせ、既存利用側の移行作業や付随ファイルの初期作成（watermark 初期化等）までをセットで完了させる。初期化ファイルの有無は移植先ごとに個別に実機確認する（「移植した」という記述は各移植先での初期化完了を保証しない）。他プロジェクトからの取り込みは単一コミットの diff だけでなく直近の履歴全体を突き合わせて漏れを確認する。フォールバックは「設定なし（空）」経路もビルド検証の対象に含め、発動時・無効化時は警告で可視化して黙らせない。警告・重要情報は出力の先頭に寄せ、切り詰めで失われない構造にする。設計に埋め込んだ暗黙の利用前提（単一リポジトリ・単発セッション等）を疑い、「最新1件だけ参照する」型の仕組みが複数対象の横断利用と組み合わさっても情報を落とさないか確認する。複数対象で同時に変更が発生した場面では、単一対象前提の特例運用（handover の一本化等）を適用せず原則（対象ごとに個別作成）へ戻す。

**なぜ有効か**: 移行・初期化・履歴突き合わせとフォールバック可視化まで含めるようにしたら、汎用化で過去に効いていた設定が黙って消える障害（2026-07-02）、空経路特有のビルド失敗、初期化漏れによる毎セッションの誤検知（2026-07-04）、複数リポジトリ横断時に前回 handover の未完了事項が無言で落ちる事故（2026-07-05）を防げるようになり、移植先での watermark 未初期化による誤検知の継続も実機確認で発見できた（2026-07-08）。

**よくある失敗パターン**: 主要シナリオだけ検証して空設定側の失敗に気づかない、仕組み本体だけ移植して初期化ファイルを作らない、`@` インクルード切れのように無効化が静かに進行する構成を放置する、単一利用では正しい設計が組み合わせで壊れることに気づかない。

## 9. 【ドキュメントは実態に揃える】

**何をするか**: 「git 管理対象/管理外」等の記述は `git status`/`git ls-files` で裏取りし、README は区切りのタイミングで最終更新コミット以降の差分を横断確認する。利用者に見える挙動が変わる変更では README 更新をタスク候補として計画に載せ、承認時に要否を確認する。手順・コマンド例は実測結果の原文を一字一句そのまま転記し、権限・対応関係のような仕組みの説明は具体例付きで書く。「X と同様に」のような参照・比喩表現は、参照先の実物の構成と突き合わせてから書く（実際には構成が異なることがある）。矛盾を見つけたら「文言を実装に合わせる」一方向でなく、実装を設計思想に合わせて直す選択肢も検討する。

**なぜ有効か**: 実態との突き合わせと転記・説明の厳密さを習慣にするようにしたら、反映漏れ（5箇所）の早期発見や、動かないコマンド例の混入（2026-07-02）、権限まわりの誤解を招く説明（2026-07-03）を防げるようになり、参照先と実際は構成が異なる不正確な表現も実物照合で訂正できた（2026-07-08）。

**よくある失敗パターン**: 「勝手に変更しない」を「更新対象外」と拡大解釈する、「大意は合っている」からとコマンド例を書き換えてしまう、参照先を確認せず「同様に」と書く。

## 10. 【委譲と指摘は作法どおりに】

**何をするか**: 同一ファイルへの連続編集は全編集完了後に1回まとめてレビューし、承認済みレビューの適用編集は再レビューをトリガーさせず完結させる（無限ループ回避）。ただし追記を重ねる作業では、追記の区切りごとの差分レビューは有効に機能する（書きかけの中間状態レビューとは区別する）。委譲時は変更内容だけでなく「対象外と判断した理由」も渡して判断の妥当性ごと検証してもらい、委譲先が理由なく kill されたら問い詰めず別モデルへ同一プロンプトで再委譲する。上位モデル前提のコマンド/スキルを Skill ツールでインライン実行するとセッションのモデルがそのまま継承されるため、設計どおりのモデルで実行したい場合は Agent ツールで `model` を明示指定する。指摘は鵜呑みにせず一次情報（`git log`・原文・`git check-ignore` 等の実機確認）で1件ずつ裏取りし、diff 全体由来のスコープ外指摘は意図した変更なら一言添えて進める。事実確認可能な主張（API 仕様等）の裏取りは指摘の採否判定にとどまらず、当初設計に無かったリスクの発見手段にもなる。上位モデルの提案に含まれる数値・閾値はプレースホルダーの可能性を疑い、転記前に具体性を検証する。

**なぜ有効か**: まとめてレビューし指摘・提案を個別検証するようにしたら、書きかけの中間状態への的外れな指摘や、正しい指摘に混ざった不正確な主張の反映（2026-07-02）、根拠のない数値の横流し（2026-07-04）を防げるようになり、裏取りの過程で設計に無かった境界迂回リスクを運用開始前に発見でき（2026-07-07）、判定基準の逆転ケースの検出や意図モデルでのスキル実行もできるようになった（2026-07-08）。

**よくある失敗パターン**: 編集の都度レビューを走らせる、複数の指摘を一括で「正しい」とみなしてまとめて反映する、上位モデルの出力を検証せず右から左に流す、モデル指定のあるコマンドをインライン実行してモデル継承に気づかない。

## 11. 【仮説より事実を優先する】

**何をするか**: 仮説と矛盾する事実が出たら実装を止めて原因究明を優先し、対策の位置づけ（根治か緩和か）は真因確定後に再評価する。SIGKILL が効かない等の異常はカーネル起因を疑い、スレッド状態（`ps -eLo`）と kernel ログ（`journalctl -k`）を見る。同一パターンの再発は単発の偶発と扱わず、チェックリスト通過を「安全が継続する保証」にしない。クロスプロジェクトの障害調査ではまず自分側の変更履歴（`git log -p`）に答えがないかを確認する。

**なぜ有効か**: 仮説に固執せず事実で再診断するようにしたら、緩和策を根治と誤認せずに済み、真因（ホストカーネル起因、2026-07-02）へ正しく到達できるようになった。

**よくある失敗パターン**: 説明できない観測結果を無視して当初仮説の対策を進める、再発している事象を毎回「実害なしの単発」としてクローズする。
