# Best Practices

> `lessons.md` の 76 項目を蒸留した高レベル原則集。
> 詳細な教訓・事例は `lessons.md` を参照。

## 1. 【合意と保全を先に】

**何をするか**: 仕様・設定の変更は必ずユーザーの合意を得てから行う。全プロジェクト共通で読まれる共有領域（`~/.claude/CLAUDE.md`・グローバルコマンド）にはプロジェクト固有の内容を書かず、編集前に元の内容を保全する。常時適用すべきルールは CLAUDE.md に置き、スラッシュコマンド化して「呼ばれない限り読まれない」形で静かに失効させない。

**なぜ有効か**: 合意・保全・配置の区別を徹底するようにしたら、無断の仕様変更、共有ファイル上書きによる他プロジェクト設定の消失事故（2026-04-25）、ルールのサイレント失効（2026-07-04）を防げるようになった。

**よくある失敗パターン**: 技術的困難を理由に黙って仕様を変える、常時ルールとたまに使うワークフローを混同して skill 化する、グローバル領域にプロジェクト固有の手順を置く。

## 2. 【許可の限界は迂回せず設計に】

**何をするか**: `dangerouslyDisableSandbox: true` を要するコマンドや許可外パスへの書き込みを実行せず、範囲外が必要なら理由を説明して許可を得る。秘密情報の実機検証は値そのものを出力させず `grep -q` やマスクで確認する。権限分類器に拒否され続ける自己監視系ファイルへの変更は自動化を諦め、最初からユーザーの手動 `!` 実行を前提に計画する。手動実行してもらうコマンドは複数行構文を避けて1行完結に寄せ、破損の疑いは `cat -A` で目視確認する。

**なぜ有効か**: 限界を迂回でなく設計で吸収するようにしたら、トークン生値の会話ログ流出（2026-07-03）や、CJK コピペ破損・ヒアドキュメント破損によるやり直し（2026-07-04）を防げるようになった。

**よくある失敗パターン**: 制約を「作業を進めるための障害」とみなして迂回する、長い複数行コマンドをそのままコピペ依頼して破損させる。

## 3. 【指示とルールを意図どおりに】

**何をするか**: 「〜まで」「〜だけ」と範囲が明示された指示は、その境界で必ず作業を止める。逆に、手順ルールの条件文を「今回のタスクは対象外」と狭く解釈して必須ステップを省かず、該当したら機械的に実行する。

**なぜ有効か**: 境界で止まり・義務を省かないようにしたら、ユーザーの介入余地を奪う先走りと、条件文の狭い解釈による必須手順の飛ばし（2026-07-04 の Plan Mode 手順スキップ）の両方を防げるようになった。

**よくある失敗パターン**: 「どうせ次もやるから」と先読みして範囲を超える、「コード編集を伴わないから対象外」と解釈してルール上のステップを飛ばす。

## 4. 【前提は実機・実経路で確認する】

**何をするか**: パッケージの有無・「完了」「実害なし」などの判断は、推測・handover・ログの記述でなく実機の裏取り（`dpkg -l`・実行結果・`podman ps`・`git status`/`git diff`）で締める。外部サービスの API 仕様・画面仕様も記憶で断定せず、エラーメッセージが返す一覧や実際の画面で確認する。検証手段そのものの落とし穴（パイプで終了コードが変わる、`git ls-files` は未追跡を含まない、`bash -n` はヒアドキュメント破損を検知しない等）も疑う。検証は本体ロジックを実際に通す経路で行い（模倣コピーの自己チェックはトートロジー）、検知ロジックの変更は既存の実データ全件に走らせて確かめる。

**なぜ有効か**: 実機・実経路での確認を徹底するようにしたら、誤った前提記録の引き継ぎ、「実害なし」とクローズした障害の継続見落とし（2026-07-02）、lint 終了コードの誤認（2026-07-03）、画面仕様の誤案内や既存データの検知漏れ（2026-07-04）を防げるようになった。

**よくある失敗パターン**: handover やログ・記憶からの推測で「存在する/しない」を断定する、本体を通さない模倣テストで「検証済み」とする。

## 5. 【ビルド資源の罠を知る】

**何をするか**: Docker/Podman のビルドキャッシュは命令文字列の一致で判定される前提で設計し、キャッシュ破棄したい箇所は `ARG` 宣言だけでなく `RUN` 内で実際に参照する。イメージ名・ステージング先のような共有ビルド資源はプロジェクトごとに名前空間で分離する。「ソース側設定の確認」と「実際に使われた成果物（ステージング先）の確認」は別工程として両方裏取りする。`Using cache` は入力が前回と同じという事実のみで、その入力が意図どおりであることは何も保証しない。

**なぜ有効か**: これらを前提に設計・検証するようにしたら、「`-b` を付けても最新化されない」バグ（2026-06-11）、複数プロジェクト交互ビルドの無言上書き（2026-07-02）、空フォールバックが安定してキャッシュヒットし続ける見落としを防げるようになった。

**よくある失敗パターン**: `ARG` 宣言だけでキャッシュが破棄されると誤解する、ソース側の設定ファイル確認だけで「ビルドに反映済み」と判断する。

## 6. 【fail-closedを既定にする】

**何をするか**: ビルドと起動のような複数ステップを1コマンドに束ねない。失敗時に古い状態へフォールバックする構成（fail-open）を避け、失敗を検知したら後続処理へ進ませない構成（fail-closed）にする。

**なぜ有効か**: build と run を分離するようにしたら、「ビルド失敗に気づかず古いイメージで起動してしまう」事故（2026-07-02）を構造的に防げるようになった。

**よくある失敗パターン**: `podman compose run --build` のような一体化コマンドに頼り、失敗検知をユーザーの目視だけに委ねてしまう。

## 7. 【取得戦略は変化頻度で選ぶ】

**何をするか**: 外部データの取得は変化頻度と通知機構の有無に合わせて設計する。変更頻度が低くレート制限のあるデータは「起動のたびのライブ取得」にせず、ビルド時に1回だけ取得してスナップショット化する。TTL が短くセットごと切り替わるデータ（CDN の A レコード等）は起動時1回の解決に頼らず、ランタイムで差分追従する。プッシュ通知が構造的に飛ばない情報（自分の活動由来の更新等）はポーリング型の確認を設計する。

**なぜ有効か**: 頻度と機構に合わせて戦略を分けるようにしたら、レート制限枯渇による起動不能（2026-07-02）と、IP ローテーション後に新規接続が全滅する障害（同日）の両方を防げるようになった。レート制限カウンタ自体も分散エッジで揺れて信頼できないため、原因追及より依存を減らす設計変更の方が確実である。

**よくある失敗パターン**: 「毎回最新を取得すべき」「起動時に1回取れば十分」「更新は通知される」という一律の直感で設計し、クォータ枯渇や許可リスト全滅・見落としを招く。

## 8. 【境界はレイヤ差を疑う】

**何をするか**: コンテナ・ネットワーク境界の設計では、あるレイヤの対策・知識が別レイヤの挙動をカバーすると思い込まない。Podman は Docker と非互換がある（ネストマウント無視等）。アプリを PID1 に置くと reap 責務の欠如でゾンビが蓄積する。パケット DROP だけでは接続候補の選定（Happy Eyeballs の IPv6 試行等）は防げず、「候補に挙げさせない」対策との両輪が要る。

**なぜ有効か**: レイヤ差を先に疑うようにしたら、効かないマウント設定への時間浪費や、ゾンビ蓄積・間欠的な接続停止（2026-07-02）を、init 導入・カーネル無効化＋DROP という多層の対策で塞げるようになった。

**よくある失敗パターン**: 「DROP しているから遮断済み」「Docker で動いたから Podman でも同じ」と、単層の対策・知識で完結したと判断する。

## 9. 【汎用化は移行まで見届ける】

**何をするか**: 仕組みの移植・汎用化・有効化では、文言でなく実装詳細と移植先の制約・規模まで確認して合わせ、既存利用側の移行作業や付随ファイルの初期作成（watermark 初期化等）までをセットで完了させる。他プロジェクトからの取り込みは単一コミットの diff だけでなく直近の履歴全体を突き合わせて漏れを確認する。フォールバックは「設定なし（空）」経路もビルド検証の対象に含め、発動時・無効化時は警告で可視化して黙らせない。警告・重要情報は出力の先頭に寄せ、切り詰めで失われない構造にする。

**なぜ有効か**: 移行・初期化・履歴突き合わせとフォールバック可視化まで含めるようにしたら、汎用化で過去に効いていた設定が黙って消える障害（2026-07-02）、空経路特有のビルド失敗、初期化漏れによる毎セッションの誤検知（2026-07-04）を防げるようになった。

**よくある失敗パターン**: 主要シナリオだけ検証して空設定側の失敗に気づかない、仕組み本体だけ移植して初期化ファイルを作らない、`@` インクルード切れのように無効化が静かに進行する構成を放置する。

## 10. 【ドキュメントは実態に揃える】

**何をするか**: 「git 管理対象/管理外」等の記述は `git status`/`git ls-files` で裏取りし、README は区切りのタイミングで最終更新コミット以降の差分を横断確認する。利用者に見える挙動が変わる変更では README 更新をタスク候補として計画に載せ、承認時に要否を確認する。手順・コマンド例は実測結果の原文を一字一句そのまま転記し、権限・対応関係のような仕組みの説明は具体例付きで書く。矛盾を見つけたら「文言を実装に合わせる」一方向でなく、実装を設計思想に合わせて直す選択肢も検討する。

**なぜ有効か**: 実態との突き合わせと転記・説明の厳密さを習慣にするようにしたら、反映漏れ（5箇所）の早期発見や、動かないコマンド例の混入（2026-07-02）、権限まわりの誤解を招く説明（2026-07-03）を防げるようになった。

**よくある失敗パターン**: 「勝手に変更しない」を「更新対象外」と拡大解釈する、「大意は合っている」からとコマンド例を書き換えてしまう。

## 11. 【委譲と指摘は作法どおりに】

**何をするか**: 同一ファイルへの連続編集は全編集完了後に1回まとめてレビューし、承認済みレビューの適用編集は再レビューをトリガーさせず完結させる（無限ループ回避）。委譲時は変更内容だけでなく「対象外と判断した理由」も渡して判断の妥当性ごと検証してもらい、委譲先が理由なく kill されたら問い詰めず別モデルへ同一プロンプトで再委譲する。指摘は鵜呑みにせず一次情報（`git log`・原文）で1件ずつ裏取りし、diff 全体由来のスコープ外指摘は意図した変更なら一言添えて進める。上位モデルの提案に含まれる数値・閾値はプレースホルダーの可能性を疑い、転記前に具体性を検証する。

**なぜ有効か**: まとめてレビューし指摘・提案を個別検証するようにしたら、書きかけの中間状態への的外れな指摘や、正しい指摘に混ざった不正確な主張の反映（2026-07-02）、根拠のない数値の横流し（2026-07-04）を防げるようになった。

**よくある失敗パターン**: 編集の都度レビューを走らせる、複数の指摘を一括で「正しい」とみなしてまとめて反映する、上位モデルの出力を検証せず右から左に流す。

## 12. 【仮説より事実を優先する】

**何をするか**: 仮説と矛盾する事実が出たら実装を止めて原因究明を優先し、対策の位置づけ（根治か緩和か）は真因確定後に再評価する。SIGKILL が効かない等の異常はカーネル起因を疑い、スレッド状態と kernel ログを見る。同一パターンの再発は単発の偶発と扱わず、チェックリスト通過を「安全が継続する保証」にしない。障害調査ではまず自分側の変更履歴に答えがないかを確認し、出自不明のファイル出現も異物と断定する前に構造的説明（bind mount による双方向の即時反映、並行セッションの進行中作業等）を先に検討する。

**なぜ有効か**: 仮説に固執せず事実で再診断するようにしたら、緩和策を根治と誤認せずに済み、真因（ホストカーネル起因、2026-07-02）へ正しく到達でき、想定外のファイル出現を環境異常と誤断せずに済むようになった（2026-07-04）。

**よくある失敗パターン**: 説明できない観測結果を無視して当初仮説の対策を進める、再発している事象を毎回「実害なしの単発」としてクローズする、想定外の観測を即「異物・環境異常」と断定する。
