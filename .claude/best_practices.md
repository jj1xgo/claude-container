# Best Practices

> `lessons.md` の 48 項目を蒸留した高レベル原則集。
> 詳細な教訓・事例は `lessons.md` を参照。

## 1. 【合意と保全を先に】

**何をするか**: 仕様・設定の変更は必ずユーザーの合意を得てから行う。全プロジェクト共通で読まれる共有設定ファイル（`~/.claude/CLAUDE.md` 等）を編集する前には元の内容を保全し、環境依存の事実は直接書かず「プロジェクト側を参照」の間接参照に委譲する。

**なぜ有効か**: 合意と保全を徹底するようにしたら、無断の仕様変更や、共有ファイル上書きによる他プロジェクト設定の消失事故（2026-04-25）を防げるようになった。

**よくある失敗パターン**: 技術的困難を理由に黙って仕様を変える、あるいは全プロジェクト共通ファイルへプロジェクト固有の内容を書き込んで元の内容を消失させる。

## 2. 【許可範囲と機密を守る】

**何をするか**: `dangerouslyDisableSandbox: true` を要するようなコマンドや、許可されていないパスへの書き込みを実行しない。範囲外が必要になったら理由を説明しユーザーに許可を得る。秘密情報（トークン等）の実機検証では値そのものを出力させず、`grep -q` の真偽判定やマスクで確認する。

**なぜ有効か**: 許可範囲と機密の扱いを先に律するようにしたら、意図しない領域への書き込みや、トークン生値を会話ログへ出力して回収不能にする事故（2026-07-03）を防げるようになった。

**よくある失敗パターン**: サンドボックス制約を「作業を進めるための障害」とみなして迂回する、存在確認のつもりで機密値をそのまま出力してしまう。

## 3. 【指示範囲で止まる】

**何をするか**: 「〜まで」「〜だけ」と範囲が明示された指示は、その境界で必ず作業を止める。次のステップが明らかでも、明示的な指示なしに先へ進まない。

**なぜ有効か**: 境界で止まるようにしたら、ユーザーが確認・タイミング調整のために介入する余地を奪わずに済むようになった。

**よくある失敗パターン**: 「どうせ次もやるから」と先読みして指示範囲を超えて作業を進めてしまう。

## 4. 【前提は実機で確認する】

**何をするか**: パッケージの有無・「完了」「実害なし」などの判断は、推測・handover・ログの記述でなく実機の裏取り（`dpkg -l`・実行結果・`podman ps`・`git status`/`git diff`）で締める。検証手段そのものの落とし穴（パイプを挟むと終了コードが変わる、`git ls-files` は未追跡を含まない等）も疑い、非自明な言語仕様は実機テストで確認する。

**なぜ有効か**: 実機確認を徹底するようにしたら、誤った前提記録の引き継ぎ、「実害なし」とクローズした障害が実は継続していた見落とし（2026-07-02）、lint 終了コードの誤認（2026-07-03）を防げるようになった。

**よくある失敗パターン**: handover やログからの推測だけで「存在する/しない」「完了した」を断定する、検証コマンドの結果自体を検証せずに信用する。

## 5. 【ビルド資源の罠を知る】

**何をするか**: Docker/Podman のビルドキャッシュは命令文字列の一致で判定される前提で設計し、キャッシュを意図的に破棄したい箇所は `ARG` を宣言するだけでなく `RUN` 内で実際に参照する。イメージ名・ステージング先のような共有ビルド資源はプロジェクトごとに名前空間で分離する。

**なぜ有効か**: この2点を前提に設計するようにしたら、「`-b` を付けても最新化されない」キャッシュ起因のバグ（2026-06-11）や、複数プロジェクトの交互ビルドが互いのイメージ・ビルドコンテキストを無言で上書きする障害（2026-07-02）を防げるようになった。

**よくある失敗パターン**: `ARG` を宣言しただけでキャッシュが破棄されると誤解する、固定パス・固定イメージ名を複数プロジェクトで共有して衝突に気づかない。

## 6. 【fail-closedを既定にする】

**何をするか**: ビルドと起動のような複数ステップを1コマンドに束ねない。失敗時に古い状態へフォールバックする構成（fail-open）を避け、失敗を検知したら後続処理へ進ませない構成（fail-closed）にする。

**なぜ有効か**: build と run を分離するようにしたら、「ビルド失敗に気づかず古いイメージで起動してしまう」事故（2026-07-02）を構造的に防げるようになった。

**よくある失敗パターン**: `podman compose run --build` のような一体化コマンドに頼り、失敗検知をユーザーの目視だけに委ねてしまう。

## 7. 【取得戦略は変化頻度で選ぶ】

**何をするか**: 外部データの取得は変化頻度に合わせて設計する。変更頻度が低くレート制限のあるデータは「起動のたびのライブ取得」にせず、ビルド時に1回だけ取得してスナップショット化する。TTL が短くセットごと切り替わるデータ（CDN の A レコード等）は起動時1回の解決に頼らず、ランタイムで差分追従する。

**なぜ有効か**: 頻度に合わせて戦略を分けるようにしたら、レート制限枯渇による起動不能（2026-07-02）と、IP ローテーション後に新規接続が全滅する障害（同日）の両方を防げるようになった。レート制限カウンタ自体も分散エッジで揺れて信頼できないため、原因追及より依存を減らす設計変更の方が確実である。

**よくある失敗パターン**: 「毎回最新を取得すべき」または「起動時に1回取れば十分」という一律の直感で設計し、クォータ枯渇や許可リスト全滅を招く。

## 8. 【境界はレイヤ差を疑う】

**何をするか**: コンテナ・ネットワーク境界の設計では、あるレイヤの対策・知識が別レイヤの挙動をカバーすると思い込まない。Podman は Docker と非互換がある（ネストマウント無視等）。アプリを PID1 に置くと reap 責務の欠如でゾンビが蓄積する。パケット DROP だけでは接続候補の選定（Happy Eyeballs の IPv6 試行等）は防げず、「候補に挙げさせない」対策との両輪が要る。

**なぜ有効か**: レイヤ差を先に疑うようにしたら、効かないマウント設定への時間浪費や、ゾンビ蓄積・間欠的な接続停止（2026-07-02）を、init 導入・カーネル無効化＋DROP という多層の対策で塞げるようになった。

**よくある失敗パターン**: 「DROP しているから遮断済み」「Docker で動いたから Podman でも同じ」と、単層の対策・知識で完結したと判断する。

## 9. 【汎用化は移行まで見届ける】

**何をするか**: 仕組みの移植・汎用化リファクタでは、文言でなく実装詳細と移植先の制約・規模まで確認して合わせ、既存利用側の移行作業（新設ファイルの用意等）を伴わせる。フォールバックは「設定なし（空）」経路もビルド検証の対象に含め、発動時・無効化時は警告で可視化して黙らせない。

**なぜ有効か**: 移行とフォールバック可視化まで含めるようにしたら、汎用化で過去に効いていた設定が黙って消える障害（2026-07-02）や、空経路特有のビルド失敗を事前に検出できるようになった。

**よくある失敗パターン**: 主要シナリオだけ検証して空設定側の失敗に気づかない、`@` インクルード切れのように無効化が静かに進行する構成を放置する。

## 10. 【ドキュメントは実態に揃える】

**何をするか**: 「git 管理対象/管理外」等の記述は `git status`/`git ls-files` で裏取りし、README は区切りのタイミングで最終更新コミット以降の差分を横断確認する。利用者に見える挙動が変わる変更では README 更新をタスク候補として計画に載せ、承認時に要否を確認する。手順・コマンド例は実測結果の原文を一字一句そのまま転記する。矛盾を見つけたら「文言を実装に合わせる」一方向でなく、実装を設計思想に合わせて直す選択肢も検討する。

**なぜ有効か**: 実態との突き合わせと転記の厳密さを習慣にするようにしたら、反映漏れ（5箇所）の早期発見や、動かないコマンド例の混入（2026-07-02）を防げるようになった。

**よくある失敗パターン**: 「勝手に変更しない」を「更新対象外」と拡大解釈する、「大意は合っている」からとコマンド例を書き換えてしまう。

## 11. 【レビューは完了後に、指摘は裏取り】

**何をするか**: 同一ファイルへの連続編集は、都度レビューせず全編集が完了してから1回まとめてレビューする。サブエージェントやレビューからの指摘は鵜呑みにせず、一次情報（`git log`・原文）で1件ずつ裏取りしてから反映する。

**なぜ有効か**: まとめてレビューし指摘を個別検証するようにしたら、書きかけの中間状態への的外れな指摘や、正しい指摘に混ざった不正確な主張（実際は別コミットで削除済み等）の反映（2026-07-02）を防げるようになった。

**よくある失敗パターン**: 編集の都度レビューを走らせる、複数の指摘を一括で「正しい」とみなしてまとめて反映する。

## 12. 【仮説より事実を優先する】

**何をするか**: 仮説と矛盾する事実が出たら実装を止めて原因究明を優先し、対策の位置づけ（根治か緩和か）は真因確定後に再評価する。SIGKILL が効かない等の異常はカーネル起因を疑い、スレッド状態と kernel ログを見る。同一パターンの再発は単発の偶発と扱わず、チェックリスト通過を「安全が継続する保証」にしない。障害調査ではまず自分側の変更履歴に答えがないかを確認する。

**なぜ有効か**: 仮説に固執せず事実で再診断するようにしたら、緩和策を根治と誤認せずに済み、真因（ホストカーネル起因、2026-07-02）へ正しく到達できるようになった。

**よくある失敗パターン**: 説明できない観測結果を無視して当初仮説の対策を進める、再発している事象を毎回「実害なしの単発」としてクローズする。
